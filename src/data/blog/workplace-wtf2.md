## 沒有規格書的開發現場

到職不到三個月，我發現這間公司的開發流程幾乎不存在所謂「流程」這件事。\
沒有 PM、沒有 QA、沒有明確規格書。需求從哪裡來？是否完成誰說得算？\
多數時候，是主管突然轉傳一份「AI 產生」的文件給我，主管自己都沒看過也看不懂。

開發會議？沒有那種東西。\
通靈 Debug 這件事我認了，開發跟需求都要通靈主管的想法？\
這種流程我這輩子沒見過。

---

## 「自己查表」文化

印象最深刻的一次，是我拿到一個新需求，從第一天就在問 Table 的 Key 該怎麼填。\
而主管只丟下一句：「自己查表。」\
問題是，Key 命名是有規則的：\
`M081` 可能代表泰國某條高速公路、`P01` 是某個設備門架、`EXF` 和 `EMF` 是什麼？我怎麼知道？

我不是在問 SQL 語法，我在問「規則」。\
但主管的反應讓我懷疑他是不是以為我連 `SELECT * FROM table_name` 都不會。\
難道我能自己隨便編一個 Key 嗎？
```sql
INSERT INTO table_name (WHATEVER_KEY_FIELD, WHATEVER_FIELD) VALUES ('M081P87NMSL', 'who cares');
```
這樣寫可以嗎？至少有資料了吧？

---

## 「先上線再修」的神奇流程

需求往往在「看到實際佈署結果」後才開始更改。\
我寫一個 Windows Service，因為沒人能告訴我 Service Name 是什麼，只好自己決定。\
結果等我**佈署完 8 個站台**後，主管才丟正確名稱給我，要求我重建、重佈署。

自動化部署？那是啥？能吃嗎？\
這裡崇尚 ***「人工部署」*** ——彷彿只有親手操作、現場跑流程，才顯得「Infra 有價值」。\
這種彼此拖台錢、拖時間的溝通方式，才能呈現所謂的「商業價值」吧？

---

## 當 AI 成了替罪羊

有一次主管提出一個新需求，不斷問我「JSON 告警訊息」會對整個系統造成什麼影響。\
我心想：我怎麼會知道？\
這套系統牽涉多個 .NET solution，我又不是能通靈跨專案的人。

於是我試著用 AI 幫忙整理資料流、畫出時序圖，讓主管能理解。\
結果換來的回應是：「不要用 AI 產文件。」\
更誇張的是，接下來他開始質疑我：「AI 產的文件你有吸收嗎？你有了解嗎？你知道 AI 畫了 Mermaid 嗎？你知道什麼是 Mermaid 嗎？」

我當然知道，否則我怎麼會交出這份文件？當我第一天當工程師？連 AI 產的內容都不會分析的小白？
更諷刺的是，要求我用 AI 產文件的人也是主管自己。\
因為某天某部門開了 AI 分享會，他覺得「我們也該試試」，\
結果當我真的照做，他又跳出來否定。\
因為主管自己也看不懂吧。

## Mentor 與「乖乖」通靈文化

老實說我的 Mentor 是誰我其實不知道\
不知道 Mentor 那怎麼知道公司原來有 Mentor 制度？在收到了 HR 的 Mentor 問卷調查才知道原來我有所謂的 Mentor\
在不知道 Mentor 是誰的情況下，我最後在意見回饋欄位上寫下了\
「有 Mentor 制度很好，但我真的不知道 Mentor 是誰，我猜是帶我吃飯的那位」

關於帶我吃飯的 Mentor 到底教過我什麼，老實說我也不記得了\
不是因為教了太多，而是根本沒帶過我過\
然而我知道 Mentor 無時無刻「叮嚀」的 2 個問題

1. 你都不看 log，你是不是不會 Debug？
2. 你是不是只會 .NET？

第二個問題我實在不知道在問什麼，原來軟體工程、設計樣式、演算法等開發能力只能用在 .NET 上？\
不論第二個問題到底想問什麼，但我深刻明白為什麼這裏稱通靈、放乖乖才叫做 Debug

**因為當你找不到原因的時候，通靈、放乖乖就是最佳解**

> 不設中斷點 Debug 看資料流，而是看 Log、放乖乖就能了解知道整個程式的走向...
> 這裡要的不是工程師，而是能卜卦算八字的通靈師或靈媒。

我能明白台灣工程師之間互相侃侃而談，開玩笑叫人放乖乖\
但是三不五時，每位工程師都在「問候」他人放乖乖了沒，這我就不懂了

*我是唯一會在其他工程師面前開乖乖來吃的工程師... 😋*
